# SyncPage 重构说明

## 1. 背景

当前同步页已经承载了太多职责：

- 配置是否完整
- 发起同步
- 选择同步模式
- 展示实时进度
- 展示上次结果
- 展示最近导出
- 展示失败项
- 展示完整日志

这些信息都放在同一页是合理的，但现在的组织方式偏散，用户需要上下扫很多块，才能回答三个核心问题：

1. 现在能不能同步
2. 上次同步结果怎么样
3. 这次同步进行到哪里了

本次重构不改成独立“概览页”，仍然放在现有 `SyncPage` 内，但把它整理成“顶部概览 + 中部操作 + 底部细节”的单页结构。

---

## 2. 当前页面问题

基于 [src/pages/SyncPage.tsx](/Users/duoshilin/duosl/sidework/biji2mdTauri/src/pages/SyncPage.tsx:42) 和当前样式体系 [src/index.css](/Users/duoshilin/duosl/sidework/biji2mdTauri/src/index.css:1)，主要问题有：

- 信息块过多，优先级不明显。
- “环境状态”“上次同步摘要”“本次同步完成”“最近导出”“失败项”“日志”彼此是平铺关系，没有主次。
- 同步前、同步中、同步后看到的内容差异不够明显。
- “上次同步摘要”和“本次同步完成”有重复。
- “最近导出”“失败项”“日志”都属于细节层，现在占比过高。
- 页面视觉虽然是极简风格，但同步页内部已经开始接近“卡片堆叠”，和设置页不够统一。

---

## 3. 重构目标

### 3.1 用户目标

- 进入页面后 3 秒内看懂当前状态。
- 配置未完成时，马上知道缺什么。
- 配置完整时，马上看到一个明确主操作。
- 同步过程中，优先看到进度和关键计数。
- 同步完成后，优先看到结果摘要，而不是整屏日志。

### 3.2 设计目标

- 保持当前产品的极简、浅色、弱装饰风格。
- 减少纵向滚动和重复信息。
- 把“概览”放到顶部，但不额外新增导航层级。
- 只保留真正高频的信息，低频细节默认收起或下沉。

---

## 4. 现有 UI 风格约束

本页重构应沿用现有界面语气，不单独发明一套视觉系统：

- 页面仍使用 `max-width: 800px` 的单列内容区。
- 保持浅背景、细边框、低饱和色。
- 标题结构沿用 `content-label + h2`。
- 优先使用“分组区块 + 细分隔”，少用厚重卡片。
- 强调色继续只承担状态提示，不做大面积装饰。
- 数字展示可以做仪表盘式分组，但形式应更接近现有 `stat-grid`，而不是复杂 BI 面板。

---

## 5. 新的信息架构

重构后的同步页建议分为 4 个区块，顺序固定：

1. 顶部概览
2. 同步操作
3. 结果与活动
4. 运行日志

### 5.1 顶部概览

职责：用最短路径回答“当前状态如何”。

建议保留 4 个指标块：

- 配置状态
- 同步状态
- 上次同步
- 最近失败

说明：

- `配置状态`：由 `hasToken + hasExportDir` 组合得出。
- `同步状态`：空闲、同步中、已取消、最近完成。
- `上次同步`：展示相对时间，如“刚刚 / 2 小时前 / 从未”。
- `最近失败`：展示最近失败数量，没有则显示“0”。

不建议第一版就放的内容：

- 本地记录总数
- 一年热力图
- 远端连接状态

原因：

- 当前接口并不完整支持这些指标。
- 这些内容会推高改造范围。
- 先把基础概览做清楚，比补图表更重要。

### 5.2 同步操作

职责：让用户在一个清晰区域完成本次同步。

这个区块只保留：

- 主按钮：开始同步 / 同步中 / 去完成配置
- 模式切换：增量 / 全量
- 模式说明
- 关键配置摘要：导出目录、分页大小
- 同步中时的取消入口

这里要做的整理：

- 将现有“环境状态卡”从独立大块降级为顶部概览的一部分。
- 将导出目录和分页大小从“隐藏上下文”提升为操作区的辅助信息。
- 未完成配置时，操作区只强调缺失项，不展示大量次要信息。

### 5.3 结果与活动

职责：展示最近一次结果，而不是所有细节都同时铺开。

这一层建议拆成两个子块：

- 上次同步结果
- 最近导出或失败项

`上次同步结果` 展示：

- 总计
- 新增
- 更新
- 跳过
- 失败
- 模式
- 是否取消

`最近导出或失败项` 的规则：

- 默认展示最近导出 5 条。
- 如果存在失败项，则失败项排在最近导出之前。
- 同步运行中不展示历史导出列表，避免和实时进度争抢注意力。

### 5.4 运行日志

职责：作为诊断层，而不是首页主视觉层。

建议：

- 日志区始终放在最底部。
- 默认折叠，或默认只显示最近 8 到 12 条。
- 增加简单筛选：
  - 全部
  - 仅错误
  - 关键节点

如果这一版不想加筛选，至少应做两件事：

- 限制默认展示条数
- 标题明确标注“日志”

## 7. 删减与合并建议

### 7.1 建议删除

- 独立的“大环境状态卡”现状写法
- 同步完成后和“上次同步摘要”高度重复的结果块

说明：

- “本次同步完成”如果保留，应该并入“上次同步结果”，而不是单独再来一个区块。

### 7.2 建议合并

- `环境状态` 合并进顶部概览
- `本次同步完成` 合并进 `上次同步结果`
- `失败项` 与 `最近导出` 合并为统一的“活动列表”区域

### 7.3 建议下沉

- 全量日志
- 路径类技术信息
- 仅在失败时才需要的诊断细节

## 8. 不同状态下的页面行为

### 8.1 未完成配置

页面优先级：

1. 顶部概览提示配置未完成
2. 操作区展示缺失项
3. 主按钮显示“去完成配置”

此时不强调：

- 最近导出
- 历史日志
- 复杂统计

### 8.2 空闲且已有历史

页面优先级：

1. 顶部概览
2. 上次同步结果
3. 最近导出
4. 日志

### 8.3 同步进行中

页面优先级：

1. 顶部概览中的“同步中”
2. 进度条与计数
3. 当前状态文案
4. 取消同步

此时隐藏或弱化：

- 历史导出列表
- 上次结果摘要

### 8.4 同步完成

页面优先级：

1. 结果摘要
2. 快捷操作
3. 最近导出
4. 日志

## 9. 组件拆分建议

如果开始实现，建议把 [src/pages/SyncPage.tsx](/Users/duoshilin/duosl/sidework/biji2mdTauri/src/pages/SyncPage.tsx:42) 拆成更清晰的展示组件：

- `SyncOverviewMetrics`
- `SyncActionPanel`
- `SyncProgressPanel`
- `SyncResultSummary`
- `SyncActivityList`
- `SyncLogsPanel`

这样做的目的不是为了“组件越多越好”，而是把状态边界拆清楚：

- 顶部概览依赖 `settings + overview + snapshot`
- 操作区依赖 `settings + snapshot`
- 结果区依赖 `overview + summary`
- 活动区依赖 `recentExports + failedItems`
- 日志区依赖 `logs`

## 12. 最终建议

这次同步页不需要“做更多”，而是应该“做更少”。

最重要的不是加很多指标，而是把页面收束成一个明确节奏：

- 先看状态
- 再做操作
- 然后看结果
- 最后才看日志

如果按这个方案实现，当前页面会更像一个“轻量仪表盘 + 操作台”，同时又不会脱离现在的 UI 风格。
